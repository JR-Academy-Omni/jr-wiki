## 为什么要有这篇

Spec-Driven Development 看上去很美——需求 → 设计 → 任务，有条不紊。但真正跑起来的团队几乎都会踩一遍同样的坑。这篇文章把最常见的七个反模式拆开来讲，附上可操作的解决方案。

---

## 反模式 1：把 Spec 当成冻结文件

**症状**：写完 `requirements.md` 就锁起来，后续发现需求变了也不敢改，因为"Spec 已经批准了"。

**为什么错**：Spec 的价值是**活文档**，不是一次性合同。需求变 → Spec 应该先变 → 代码跟着变。把 Spec 冻结意味着你只是把混乱从代码层转移到了文档层——代码在按 Spec 跑，但 Spec 已经不对了。

**解决方案**：

```markdown
<!-- 在 requirements.md 顶部加版本记录 -->
# 用户认证需求
> v1.0 2026-05-10 初版
> v1.1 2026-06-15 去掉手机号登录（产品决策），加企业 SSO 要求

## 功能需求
...
```

把 Spec 文件纳入正常 code review 流程。需求变更 = PR 先改 Spec，review 通过后再让 Agent 更新代码。

## 反模式 2：过度规格化实现细节

**症状**：`requirements.md` 里写"使用 bcrypt 做密码哈希，rounds 设为 12"，而不是"密码必须安全存储，不能明文"。

**为什么错**：规格化**结果**，不规格化**手段**。过度指定实现细节有两个害处：
1. Kiro 的 AI 能力在实现层比你强（它知道最新的安全实践），限制它反而产出更差
2. 技术方案变了（比如从 bcrypt 换 argon2），Spec 得跟着改，维护成本翻倍

**对比示例**：

```markdown
# ❌ 过度规格化（指定手段）
WHEN 用户注册 THE SYSTEM SHALL 用 bcrypt(rounds=12) 哈希密码后存入 users.password_hash 字段

# ✅ 正确写法（指定结果）
WHEN 用户注册 THE SYSTEM SHALL 以不可逆的方式安全存储密码，确保即使数据库泄露也无法还原明文
```

EARS 语法的核心是描述系统**行为**和**约束**，而不是实现路径。

## 反模式 3：小功能不写 Spec

**症状**："这个改动太小了，加个字段而已，不用走 Spec 流程。"

**为什么错**：大功能的 bug 容易在 code review 时发现，小功能的假设却容易悄悄溜过去。"删除记录"是软删除还是硬删除？"禁用用户"是不让登录、还是连数据都隐藏？这类歧义在小功能里最多见。

一个简短的 Spec 比事后修 bug 便宜得多：

```markdown
<!-- .kiro/specs/soft-delete-spec/requirements.md -->
# 软删除功能

WHEN 管理员删除文章 THE SYSTEM SHALL 标记 deleted_at 字段而非物理删除记录
WHEN 普通用户访问被删除的文章 THE SYSTEM SHALL 返回 404（不暴露内容存在）
WHEN 管理员查看文章列表 THE SYSTEM SHALL 提供"含已删除"的筛选选项
```

三条需求，两分钟写完，消除了所有歧义。

## 反模式 4：忽视验收标准（Acceptance Criteria）

**症状**：`requirements.md` 写得很详细，但 `tasks.md` 里的任务没有验收标准，只有"实现 XX 功能"。

**为什么错**：验收标准是整个 Spec 流程里价值最高的部分。没有它，Agent 完成任务后你无法判断对不对——因为"对"的定义从来没说清楚。

**好的验收标准长这样**：

```markdown
## Task: 实现登录限流

实现步骤：
- [ ] 在 Redis 里记录每个 IP 的失败登录次数
- [ ] 失败 5 次后锁定账号 15 分钟

验收标准：
- [ ] 用同一 IP 连续失败 5 次后，第 6 次返回 429 + "账号已锁定，15 分钟后重试"
- [ ] 锁定期间成功验证的请求也返回锁定提示（防止账号枚举）
- [ ] 锁定计数在 15 分钟后自动清零，重新允许尝试
- [ ] 单元测试覆盖边界：第 4/5/6 次尝试的不同响应
```

每个任务的验收标准是可以直接转成测试用例的——让 Agent 写测试时直接引用这些标准，测试和需求天然对齐。

## 反模式 5：不用歧义检测，自以为需求清晰

**症状**：跳过 Kiro 的歧义检测步骤，直接从 requirements.md 让 Agent 生成 design.md。

**为什么错**：你认为清晰的需求，AI 可能有三种不同的理解，你却不知道它选了哪一种。Kiro 的歧义检测会在生成 design.md 之前，把模糊点以二选一的问题形式抛给你：

```
❓ 关于"删除订单"需求，有两种理解：
   A. 软删除：保留数据，订单状态变为"已取消"，在管理后台可查
   B. 硬删除：从数据库移除，不可恢复

你的意图是哪种？
```

花 30 秒回答这些问题，能避免实现完成后才发现方向完全错了的情况。

## 反模式 6：Spec 和代码失去同步

**症状**：功能上线了，但 `.kiro/specs/` 里的 `requirements.md` 还是三个月前的版本。

**为什么错**：Spec 失活后，下次让 Agent 扩展这个功能时，它会基于过时的 Spec 做设计，产生和现有代码相矛盾的方案。

**系统性解决方案**：用 Agent Hook 在代码合并时自动提醒：

```json
// .kiro/hooks/spec-sync-reminder.kiro.hook
{
  "title": "提醒同步 Spec 文件",
  "when": {
    "type": "fileSaved",
    "patterns": ["src/**/*.ts", "src/**/*.py"]
  },
  "instruction": "检查当前修改的文件是否有对应的 .kiro/specs/ 目录。如果有，列出 spec 里可能和改动不符的需求条目，提示开发者更新。不要自动修改 spec 文件。"
}
```

这个 Hook 不会自动改 Spec（因为 Spec 变更需要人工审查），但会在你保存代码时提醒你检查。

## 反模式 7：无条件信任 AI 生成的代码

**症状**：Kiro 完成 tasks.md 里的所有任务后，直接合并，没有人工审查。

**为什么错**：AI 生成的代码有一个危险特质：**在错误的时候也显得很自信**。代码格式正确、逻辑看似合理，但可能在某个边界条件下会出问题——而这个问题只有真正理解业务的人才能发现。

**高风险场景需要人工审查**：

```
风险等级 HIGH → 必须人工 review：
- 涉及金钱计算（舍入误差、货币换算）
- 权限/认证逻辑（越权漏洞不会触发测试失败）
- 数据库 schema 变更（跑了 migration 才发现设计有问题）
- 第三方 API 集成（文档理解偏差只有真实调用才能发现）

风险等级 MEDIUM → 跑测试后 review：
- 业务逻辑（让 Agent 先写测试，测试通过后再 review 代码）
- 性能相关（看 query 计划和索引使用）

风险等级 LOW → 自动合并（需配合严格的 Hook）：
- 文档更新
- 格式化/lint 修复
- 测试文件生成（人工补充边界用例）
```

把这个风险矩阵写进 `.kiro/steering/review-policy.md`，让 Agent 在生成任务时自动标注风险等级。

## 快速自检清单

开始一个新 Spec 前，对照检查：

```
□ requirements.md 描述的是"系统应该做什么"，而不是"怎么实现"
□ 每条需求都有对应的验收标准（可转化为测试用例）
□ 经过了歧义检测步骤，所有两义性问题都已明确回答
□ 功能大小不影响是否写 Spec（3 行的 Spec 也比没有强）
□ Spec 文件纳入 Git，和代码同步演进
□ tasks.md 里高风险任务已标注，完成后会有人工 review
```

踩过这些坑的团队通常在第 3-4 周开始发现 Spec-Driven 的真正价值：不是快了多少，而是**返工少了多少**。
