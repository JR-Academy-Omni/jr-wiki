📅 入职第一天，别人让你"把东西放 GitHub 上"

L0 · 先搞懂这俩是啥，再动手

先讲个你一定会遇到的场景第一天上班，同事甩给你一句："你把改好的课程大纲放到 GitHub 上，开个 PR 我 review 一下。"
你心里 OS：GitHub 是啥？放？PR 又是啥？……别慌，这一级我先把这几个词给你说成人话。

🎭 先认识俩主角：git 和 GitHub

很多人一上来就把这俩搞混。其实特别简单，一个是工具，一个是网站：

git
一个工具 · 装在你电脑上
帮你记录每次改了啥、还能随时反悔退回去。你就把它想成"整个项目的修订历史 + 后悔药"。它在你电脑里默默干活，平时你都不用直接碰它。
GitHub
一个网站 · 在网上（云端）
大家把项目放上网，一起看、一起改的地方。你可以把它理解成"公司的共享网盘"，只不过它专门给代码和文档用，还带完整历史记录。
💡 一句话记住：git 是你电脑上的"后悔药工具"，GitHub 是网上"大家一起放东西的地方"。你用 git 在自己电脑上记录，再把东西放到 GitHub 上跟同事共享。

🕰 git 最值钱的地方：每改一次，都给你存一个"版本"

普通文件你改了存了，旧的就没了，想反悔只能哭。git 不一样：你每提交一次，它就悄悄给整个项目拍一张快照，存成一个版本，还记下"谁、啥时候、改了啥"。所有这些版本都躺在 GitHub 云端——哪天你手抖删错、改砸了，随时能翻回到以前任何一个版本。

这就是为什么公司的东西放 git / GitHub 上最安全——它不是"覆盖式保存"，而是"每一版都留底、全存云端"。你几乎不可能真把东西弄丢。

🗯 AI 老爱跟你蹦这些词，我先翻成人话

下面这些英文词，你早晚会听到 Claude 或同事说。你不用会用，先能听懂就行——记我这版人话，别去看那些绕的专业解释：

repo仓库 repository就是一个项目的文件夹（连同它的全部历史）。比如咱们的网站项目就是一个 repo。
clone克隆第一次把网上的项目整个拷到你电脑。一个项目只做一次，之后就用 pull 更新。（下一节细讲）
commit提交存个档、拍张快照，顺手写一句"我这次改了啥"。像游戏存档点。
push推送把你电脑上的改动"交上去"到 GitHub。交了，同事才看得到。
pull拉取把 GitHub 上别人的新改动"取下来"到你电脑。开工前先取，免得用的是旧的。
branch分支你自己的一份副本，随便改，不影响别人。（L2 细讲）
main主分支 / 旧称 master正式版那条线。有些老项目叫 master，一个意思。默认别直接动它。
PRPull Request"我改完了，请你看一眼再合并"的一个申请。给同事 review 用的。
merge合并PR 通过后，把你的改动正式并进 main。到这一步你的活儿才算"进了正式版"。

看不懂没关系，这张表你随时能翻回来。下面真正干活的时候，你会发现——这些你一个都不用自己敲。

🔌 最最开始：怎么把项目弄到你电脑上（一个项目只做一次）

新人第一个真问题"项目在 GitHub 网上，我电脑里啥都没有，我改个锤子？"
对，所以你上手的第一件事不是改文件，是先把整个项目从网上拷一份到你电脑——这个动作叫 clone（克隆）。同一个项目这辈子只 clone 一次，之后每天开工用 pull 更新就行。

所以你入职的起步三步是：
1️⃣ 注册 / 登录 GitHub 账号，把账号给同事，请他把你拉进项目（不然你没权限）。
2️⃣ 让 Claude 帮你 clone：对它说下面这句。
3️⃣ clone 完，项目就在你电脑上了，以后才谈"改文件"。

第一次帮我把 XX 项目 clone 到我电脑上

🗣 项目到电脑后，日常干活只要对 Claude 说两句话

上面那堆词，全是 Claude 的活儿。你只管用大白话告诉它"我开工了 / 我弄完了"，它替你把那些命令跑掉：

开工时帮我拉最新，开个今天的分支
弄完时帮我提交，开个 PR @我的 review 人

🗺 你改的东西，要"走 4 站"才算交出去

你可能以为"我改完保存就完事了"。其实你的东西要走完下面这趟车，才算真的交到公司手里——而且最容易栽的坑就在中间那条红线：

注意中间那条红色虚线：前两站（改、提交），东西还窝在你自己电脑里，谁都看不见；一定要坐到第 3 站「推送 push」，过了红线，同事和老板才看得到。这就是新人最常踩的坑——以为保存了 = 交了。

✅ push 完心里没底？这样确认它真上去了

新人 push 完最常问"它…成了吗？"。别靠感觉，去 GitHub 网页眼见为实：

打开浏览器 → 进你们项目的 GitHub 页面 → 看到你刚写的那句改动说明、时间是"几秒前 / just now"、作者是你 → ✅ 成了。
看不到？大概率是没 push（只 commit 了）或没开 PR。直接问 Claude："我刚才的改动 push 上去了吗？帮我确认下。"
✅ 这一级你不用全记住。只要：知道 git=工具 / GitHub=网站、第一次要 clone、那张黑话表能翻回来看，再让 Buddy 陪你跑通一次"两句话"，就过关了。

L1 · 本地 vs 云端（最重要的一级）

这事八成会发生在你身上你周五下午改完课程大纲，Ctrl+S 存好，信心满满跟同事说"我弄完了，你打开看看"。
结果同事打开一看——啥都没变，还是老样子。你俩面面相觑。
不是你电脑坏了，也不是同事眼神不好。是 git 有个让每个新人都栽一次的设定 👇

同一份大纲，其实有两份：一份在你电脑里，一份在 GitHub 上。
它俩不会自己同步。

🖼 看这张图就懂了

所以你那次"白存了"，是因为东西只待在左边（你电脑），你没做那个绿色的 push 把它交到右边（GitHub）。同事看的是右边——右边没有，他当然看不到。

🚗 为什么不能像网盘那样自动同步？

你肯定想问：网盘改完自己就传上去了，多省心，git 干嘛非要我手动？
因为 git 觉得"自动"太危险——万一你和同事同时改同一个文件，自动同步会"咣"一下把其中一个人的活儿盖没了，还没人发现。所以它故意改成手动：你不亲口说一声"传上去"，它就一直憋着，搁你电脑里不动。多按一下，换来谁的活儿都不会被悄悄覆盖。

📁 普通网盘
改完自动同步，省心
⚠️ 但可能默默盖掉别人的改动
🔧 git（手动）
你说"传"才传、"取"才取
✅ 每一步看得见，不误伤队友

🎮 自己点点看：东西到底去哪边

光看不够，你点几下就有感觉了——看那个"大纲.md"文件，在你点按钮时怎么从左边跑到右边：

💻 你的电脑
草稿桌 · 只有你看得到
push →← pull
☁️ GitHub
档案室 · 全公司看得到
👆 先点①，看文件出现在哪边；再点②，看它怎么跑到右边去。

⚠️ 两个所有人都犯过的误会（点开看）

误会① "我 Ctrl+S 保存了，就等于交上去了"
点一下展开
❌ 保存只是把东西留在你自己电脑里。别人、老板、线上网站，统统看不到。一定要 push（推上去），才算真交。记住：保存在电脑 ≠ 交到公司
误会② "我电脑上看到的，肯定是最新版"
点一下展开

📅 当你发现，这个项目不止你一个人在改

L2 · 怎么跟人一起改

第二个坎用了几天你会意识到：这项目好多人都在改。问题立刻来了——大家都往同一份文件上招呼，不就乱套、互相覆盖了吗？
git 的解法挺聪明：每个人先各自复制一份去改，谁也不碰谁，改好了再合到一起。这份你自己的副本，就是 L0 表里那个词——"分支"。

🌳 分支：先在自己的副本上改，改好再合回正式版

黑线是 main（公司的正式版）。你不直接在它上面改，而是从它岔出一条紫线（你的分支）去改；改好了，发一个 PR 请同事看一眼，没问题就合回黑线。这样大家各改各的，最后干干净净汇到一起。

💥 "冲突"听着吓人，其实就是 git 在问你"听谁的"

你迟早会看到"冲突 conflict"这个词，第一反应肯定是"完了，我把项目搞坏了"。真没有。冲突就是上图这么回事：你和同事不约而同改了同一行，git 一看傻眼了——"你俩写的不一样，我不敢替你们拍板"，于是它停下来举手叫人。这不是报错，是 git 在等你拿主意。叫上同事一起定一下就好。

🔁 我开了 PR，然后呢？（很多人卡在这）

真实困惑"PR 开完了，我是不是就没事了？要一直刷新等吗？"——不是干等，它是个小循环：

所以开完 PR 你不用干等：同事看完会 @ 你。如果他说"这里改一下"，你在同一个分支改了、再 push 一次，PR 会自动更新，不用重开。来回几轮直到他点通过、合并(merge)，你这次活儿才真进了正式版。被要求改不丢人，这就是正常流程。

📌 名词混了就回来瞄一眼：main = 正式版 · 分支 = 你的副本 · 提交(commit) = 存个档 · PR = 请人看了再合并 · 合并(merge) = 把你的改动并进正式版。

L3 · 出事不慌

总有这么一天你会突然看到满屏红字，心里咯噔一下——"我是不是把公司项目搞炸了？"
先深呼吸。99% 的情况你啥也没炸。真正会把事情搞大的，往往是慌乱中乱点。所以你只要记住一件事：出事别自己硬刚，叫人。

🚦 看到报错，往哪条路走？

就这么简单：慌的时候走右边。把屏幕整个截下来，发工作群，写一句"我刚做了 XX，冒出来这个"，等人帮你看。或者直接问 Claude"这是啥意思、我该怎么办"。千万别走左边自己乱试。

🚫 3 条红线（背下来，比啥都管用）

❌
不在 main（正式版）上直接改
❌
看到报错 / 冲突别硬来，截图发群叫人
❌
别人教你的"强行覆盖（force push）"、"删分支"——没人盯着别自己点
✅
迷路了就问 Claude："我现在在哪个分支？改了什么？帮我看看"

🙅 还有：这些东西千万别往 git 里塞

GitHub 是给文字 / 代码 / 文档用的，而且一旦 push 上去，历史里就洗不掉了（哪怕你后来删了，旧版本还留着）。所以这几类东西不要往里放：

🔑
密码 / 密钥 / token / API key —— 等于公开挂网上，最危险。要用问同事怎么走安全方式。
🧑
客户名单 / 学员隐私 / 身份证 / 简历 / 薪资 —— 个人信息进 git 是合规事故，放该放的系统（如 Notion / 后台）。
🎬
大文件：视频 / 大图 / 几百 MB 的素材 —— git 扛不动会拖垮整个项目，用网盘 / S3 放。
🤔
不确定"这个能不能放"？—— 先别 push，问一句。push 前问一句 = 0 成本；push 错了再清 = 大麻烦。
